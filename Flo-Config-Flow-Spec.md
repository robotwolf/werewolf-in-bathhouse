# Flo Config Flow Spec

This doc is the first concrete spec for `Flo`, the future layer that governs flow between distinct `Ginny` layout regimes.

`Flo` is not implemented yet.

That is intentional.

This exists so we can stop talking about Flo like a prophecy written on a wet napkin and start treating her like planned architecture.

## Purpose

`Ginny` assembles one local layout regime.

`Flo` will eventually decide:

- which layout regimes exist in a run
- how they connect
- what counts as forward, backward, optional, locked, or wrong
- how the player transitions between them

`Mason` and `Butch` remain downstream concerns.

## Mental Model

The larger world is a graph of layout configs.

Each config node is a locally coherent regime generated by `Ginny`.

Examples:

- `Bathhouse_PublicCore`
- `SecondFloor_PrivateCubicles`
- `ServiceSpace`
- `WrongSpaceIntrusion`
- future:
  - `MoonPeak_ShiftedPublicCore`
  - `ParkingLot_Exterior`
  - `RVPark_MainLoop`

`Flo` owns the graph.

`Ginny` owns what each node becomes internally.

## Responsibilities

### Flo should own

- config ids
- parent/child or graph relationships between configs
- transition permissions
- transition directionality
- return-path rules
- optional vs required config presence
- future "this path lies to the player" routing logic

### Flo should not own

- local room placement
- primitive construction
- room dressing
- NPC simulation internals

If a concern only exists inside one local layout regime, it probably belongs to `Ginny`, not `Flo`.

## Core Terms

### Layout Config

A named local generation regime.

Examples:

- `Bathhouse_PublicCore`
- `SecondFloor_PrivateCubicles`

### Transition Node

A room, doorway, stair, or other authored handoff point that leads from one config to another.

Today, the stair room already carries the first stub of this idea:

- `TransitionType = ConfigHandoff`
- `TransitionTargetConfigId = SecondFloor_PrivateCubicles`

### Flow Edge

A directed relationship between two layout configs.

Examples:

- `Bathhouse_PublicCore -> SecondFloor_PrivateCubicles`
- `SecondFloor_PrivateCubicles -> Bathhouse_PublicCore`
- future:
  - `Bathhouse_PublicCore -> WrongSpaceIntrusion`

## First Intended Use

The first real Flo target should be:

1. `Bathhouse_PublicCore`
2. `SecondFloor_PrivateCubicles`

That handoff should be mediated by one explicit stair transition.

The second floor should feel maze-like, but still be generated as its own local regime with its own rules.

## First Version Constraints

When Flo is first implemented, keep her on a short leash:

- one bathhouse public-core config
- one upstairs cubicle config
- one explicit transition type
- one honest return path
- no impossible-space substitutions yet
- no moonrise rerouting yet

That keeps the first handoff readable and debuggable.

## Proposed Data Shape

These names are suggestions, not commandments.

### `UFlowConfigProfile`

Represents one layout regime entry.

Could contain:

- `ConfigId`
- reference to `UGinnyLayoutProfile`
- required / optional presence
- weight or occurrence rules
- future gating tags or phase conditions

### `FConfigTransitionRule`

Represents one edge between two configs.

Could contain:

- `FromConfigId`
- `ToConfigId`
- `TransitionType`
- `bAllowReturn`
- `bRequired`
- `bVisibleAsNormalArchitecture`
- future:
  - phase gating
  - illusion allowances
  - NPC restrictions

### `UFloFlowProfile`

Represents one whole config-flow graph for a run.

Could contain:

- root/start config
- list of config profiles
- list of transition rules
- optional progression constraints

## Runtime Shape

Very rough intended runtime behavior:

1. Flo selects which configs are active.
2. Flo resolves their graph.
3. Each config is handed to `Ginny` for local generation.
4. Transition rooms become the handoff edges.
5. The player traverses the larger structure through Flo-managed transitions.

## Relationship To Existing Work

Current stair metadata is already the right seam.

That means we do **not** need to throw away the stair work when Flo arrives.

Instead:

- today: stair is a transition-ready room that currently terminates
- later: stair becomes the handoff into a real second config

That is exactly the kind of incremental architecture we want.

## Debug Expectations

When Flo exists, she should produce a summary just as clearly as Ginny does.

Minimum future summary:

- configs active this run
- root config
- transition edges selected
- which transitions are required vs optional
- any failed or gated transitions

If Flo can’t explain herself, she’s just a haunted router.

## Deferred On Purpose

This spec does **not** yet implement:

- moonrise rerouting
- false-return-space substitutions
- subway-car wrong-space intrusions
- steam-burst teleport wipes
- config-level NPC ownership

All of those can sit on top of the same flow model later.

## Acceptance Criteria For Flo Phase 1

We should not implement Flo until we are ready to meet these criteria:

1. one public-core config can hand off to one upstairs config
2. the handoff is visible in data, not magic hardcoding
3. the player can return cleanly
4. Ginny summaries still make sense per local config
5. Flo can summarize the config graph chosen for the run
