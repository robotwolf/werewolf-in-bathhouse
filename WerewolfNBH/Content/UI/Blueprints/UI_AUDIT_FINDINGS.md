# UI Audit Findings

Date: 2026-03-13
Project: WerewolfNBH

## Verified Current State

The following audit was run through the editor bridge against the saved widget assets:

- `WBP_SteamMeter`
- `WBP_MeterTotemBase`
- `WBP_HUDInteractionPrompt`
- `WBP_ObjectiveLine`
- `WBP_DoorStateBadge`
- `WBP_AccessTagBadge`
- `WBP_HUDRoot`

## Confirmed Findings

### 1. The child widgets exist

`WBP_SteamMeter` and `WBP_MeterTotemBase` both contain:

- `Image_Backplate`
- `Overlay_Root`
- `ProgressBar_Fill`
- `Text_Label`
- `Text_Value`

So if you do not see a label/value/progress widget in the asset, the issue is not that the scaffold failed to create them.

### 2. The child widget functions do not exist yet

`WBP_SteamMeter`, `WBP_MeterTotemBase`, `WBP_HUDInteractionPrompt`, `WBP_ObjectiveLine`, `WBP_DoorStateBadge`, and `WBP_AccessTagBadge` all audited with:

- `FunctionGraphs=(none)`

That means names like:

- `SetMeterData`
- `SetPrompt`
- `SetObjectiveLine`
- `SetDoorState`
- `SetAccessTag`

were never autogenerated by the scaffold. Those were always intended as manual Blueprint functions.

### 3. WBP_HUDRoot already has manual functions now

The audited `WBP_HUDRoot` currently contains these function graphs:

- `SetFade`
- `Update Prompt`
- `UpdateDoorAndAccess`
- `UpdateObjective`
- `UpdateSteam`

So the current mismatch is:

- parent HUD functions exist
- child widget helper functions do not

If `UpdateSteam` is trying to call `SteamMeterRef.SetMeterData`, that call will fail until `SetMeterData` is created in `WBP_SteamMeter`, or until `UpdateSteam` is rewritten to drive the child controls directly.

## Instruction Corrections Applied

The UI docs were corrected so they now explicitly say:

- the scaffold created widget trees and placeholder controls
- the graph functions are still manual Blueprint work

Updated docs:

- `UI_BLUEPRINT_HOOKUP_CHECKLIST.md`
- `UI_BLUEPRINT_NODE_RECIPES.md`
- `UI_ONE_EVENING_STARTER_PATH.md`

## Wipe Framework

A separate standalone widget now exists:

- `WBP_ScreenWipeFramework`

This was created as a separate overlay instead of being injected into `WBP_HUDRoot`, because `WBP_HUDRoot.uasset` already had local edits and should not be overwritten blindly.

The wipe framework currently includes these layers:

- `FX_WipeBaseBlack`
- `FX_IrisMask`
- `FX_SteamWipeFront`
- `FX_SteamWipeBack`

This is a framework only. The actual iris mask material and steam wipe materials still need to be authored and driven from Blueprint or Sequencer.

## Recommended Next Fix

For the fastest path forward:

1. Add `SetMeterData` to `WBP_SteamMeter`
2. Add `SetPrompt` to `WBP_HUDInteractionPrompt`
3. Add `SetObjectiveLine` to `WBP_ObjectiveLine`
4. Add `SetDoorState` to `WBP_DoorStateBadge`
5. Add `SetAccessTag` to `WBP_AccessTagBadge`

After that, the existing `WBP_HUDRoot` update functions should have something valid to call.
